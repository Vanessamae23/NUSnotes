Type Erasure

Java takes a code sharing approach, instead of creating a new type for every instantiation, it chooses to erase the type parameters and type arguments during compilation (after type checking, of course). Thus, there is only one representation of the generic type in the generated code, representing all the instantiated generic types, regardless of the type arguments.

➡️

Note that each type parameter Sand Tare replaced with Object. If the type parameter is bounded, it is replaced by the bounds instead (e.g., If T extends GetAreable, then T is replaced with GetAreable)

Integer i = new Pair<String,Integer>("hello", 4).getSecond(); ⇒ Integer i = (Integer) new Pair("hello", 4).getSecond(); because the return type of getSecond() is T which is now an Integer object;

Even if it seems like they have different method signatures so “overloading” seems alright, the compiler will complain since both will be erased to void foo(Queue c) so compilation error occurs

😱 Generics are invariant but Arrays are covariant

// create a new array of pairs
// THE BELOW IS HYPOTHETICAL SO TECHNICALLY IT WILL NOT COMPILE
Pair<String,Integer>[] pairArray = new Pair<String,Integer>[2];

// pass around the array of pairs as an array of object
Object[] objArray = pairArray;

// put a pair into the array -- no ArrayStoreException!
objArray[0] = new Pair<Double, Boolean>(3.14, true);

// After type erasure ----------------------------------------
// create a new array of pairs
Pair[] pairArray = new Pair[2];

// pass around the array of pairs as an array of object
Object[] objArray = pairArray;

// put a pair into the array -- no ArrayStoreException!
objArray[0] = new Pair(3.14, true);
// The above is fine, compiler won't complain
  • The runtime has no idea about the type arguments that are passed to Pair class so since Array is covariant Object[] objArray = pairArray; this is okay! It checks that we have an array of Pair and we are just putting another Pair ⇒ This will cause heap pollution (a term that refers to the situation where a variable of a parameterized type refers to an object that is not of that parameterized type).
  • But when try to put it another Pair of different type, there will be NO ArrayStoreExceptionRemember: We would not, however, get an exception when we try to put a pair of double and boolean, into an array meant to store a pair of string and integer
  • The problem is when we try to retrieve the data
// getting back a string?  -- now we get ClassCastException
String str = pairArray[0].getFirst();

This will cause us to get ClassCastException which is a Runtime Exception

Side reason : (TLDR ⇒ read the heading) An array is what is called reifiable type -- a type where full type information is available during run-time. It is because Java array is reifiable that the Java run-time can check what we store into the array matches the type of the array and throw an ArrayStoreException at us if there is a mismatch. Java generics, however, is not reifiable due to type erasure.

😱 Using static methods or fields for Generics

class Pair<T> {
    static int x = 1;

		//This is not allowed
    static T foo;

		// This is not allowed
    static T bar(T t) {
        return t;
    } 

		// This is ok!
    static <T> T bark(T t) {
        return t;
    }
}
  • static int x = 1 : After type erasure, Pair<String> and Pair<Integer> will be compiled to Pair⇒ so they will share the SAME x although it might seem like they are two different classes
  • static T foo and static T bar(T t) are NOT ALLOWED and will generate compilation error because there will only be one copy of foo shared by both Pair<String> and Pair<Integer> and the compiler doesn’t know what
  • static <T> T bark(T t) ⇒ This is okay because we need to declare a type when we want to use the method so the T here does not refer to the T in the class Pair
